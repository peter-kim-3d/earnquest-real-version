## 0. 핵심 정의

### 0.1 포인트의 정체

| 관점 | 포인트란 |
|------|----------|
| **아이에게** | "돈"이 아니라 **게임 토큰 / 선택 티켓** |
| **부모에게** | "시간/돈"이 아니라 **노력(습관)의 상대적 크기** |
| **제품에게** | **단일 통화(One Currency)** — 모든 카테고리를 한 지갑에서 운영 |

### 0.2 절대 금지 원칙

| 금지 항목 | 이유 |
|----------|------|
| UI/카피에 `1포인트 = n분 / n원` 공식 노출 | 노동=돈 프레임 강화 방지 |
| 현금 환산형 Savings | 금융 교육이 아닌 습관/선택 교육이 목적 |
| 포인트 "구매" 표현 (Screen 카테고리) | 환율 암시 방지 |

## 1. Store 철학

### 1.1 Store의 목적

Store는 **"상점"이 아니라 "선택 연습 공간"**입니다.

**목적:**
- 즉시 만족 vs 장기 목표를 균형 있게 선택하는 연습
- 포인트 환율 없이 "이 선택이 어떤 노력 크기인지" 감각 형성
- 소비가 아닌 **의사결정 능력** 개발

### 1.2 Store IA (정보 구조)

**아이 Store 기본 노출 순서:**

```
┌─────────────────────────────────────┐
│  1. Goals (장기 목표)               │  ← 가장 위, 가장 크게
│     "큰 꿈을 향해 모으는 중"         │
├─────────────────────────────────────┤
│  2. Rewards (일반 리워드)           │  ← 중간
│     Autonomy + Experience 통합      │
├─────────────────────────────────────┤
│  3. Screen Time (스크린)            │  ← 아래쪽
│     "오늘의 스크린 예산"             │     (접근 마찰 의도적)
└─────────────────────────────────────┘
```

**원칙:** 스크린을 "금지"하지 않고, **접근 순서/노출 비중**으로 균형을 만든다.

### 2.2 Screen Time (스크린 타임)

#### 핵심 변경: "구매"가 아닌 "예산 사용"

**V1.0 문제:**
```
- 아이가 포인트로 스크린 시간을 "구매"
- 숨겨진 환율 존재 (예: 10p = 15분)
- 아이가 역산 가능 → 환율 숨기기 원칙과 충돌
```

**V1.1 해결:**
```
- 스크린 타임은 포인트와 "분리된 예산 시스템"
- 주간 스크린 예산은 "기본 제공" + "보너스 획득" 구조
- 포인트로 "구매"하는 것이 아니라, 퀘스트 완료 시 "보너스 시간 획득"
```

#### Screen Time 규칙

| 요소 | 설명 |
|------|------|
| **기본 예산** | 부모가 설정한 주간 기본 스크린 시간 (예: 주 3시간) |
| **보너스 시간** | 특정 퀘스트 완료 시 추가 획득 (예: +15분) |
| **일일 상한** | 하루 최대 사용 가능 시간 (예: 1시간) |
| **쿨다운** | 연속 사용 방지 (예: 30분 사용 후 30분 휴식) |

#### 아이 UI 표시

```
┌─────────────────────────────────────┐
│  🎮 오늘의 스크린 타임               │
│                                     │
│  ████████░░░░░░░░  45분 / 60분      │
│                                     │
│  이번 주 남은 시간: 2시간 15분       │
│                                     │
│  [사용하기]                         │
└─────────────────────────────────────┘
```

**핵심:** 포인트 숫자가 아닌 **"분" 단위 게이지**로만 표시

### 2.3 Rewards (일반 리워드)

#### V1.0 Autonomy + Experience 통합

**V1.1에서는 두 가지 상태로만 구분:**

| 상태 | 설명 | 예시 |
|------|------|------|
| **즉시 사용** (Instant) | 구매 즉시 효력 발생 | 간식 선택권, 메뉴 결정권 |
| **승인 필요** (Approval) | 부모 확인 후 사용 | 외식, 늦잠, 특별 활동 |

#### Experience형 리워드 특별 규칙

"예약이 필요한" 리워드 (외출, 특별 활동 등):

```
아이 요청 → 부모 일정 확정/거절 → 확정 시 "예약됨" 표시
```

**신뢰 장치:**
- 부모 미응답 (48시간): **자동 100% 환불**
- 부모 거절: **즉시 100% 환불 + 사유 표시**
- "샀는데 못함" 좌절을 **구조적으로 차단**

#### Trust 시스템 (V1.1에서 제거)

**V1.0:**
```
"중간 위험: Trust 조건 + 안전 가드 만족 시 자동"
→ Trust 조건이 무엇인지 정의 없음
```

**V1.1:**
```
Trust 시스템 제거
모든 리워드는 "즉시" 또는 "승인 필요" 둘 중 하나
Trust 기반 자동화는 V2에서 데이터 축적 후 도입
```

### 2.4 Goals (장기 목표)

#### 정의

Goals는 "저축"이 아니라 **Goal Vault (큰 목표 바구니)**:
- 큰 목표(레고, 자전거, 특별 여행 등)를 설정
- 포인트를 모아가는 진행 트랙
- **현금/원/달러와 완전 단절**

#### 아이 UI 표시

```
┌─────────────────────────────────────┐
│  🎯 나의 목표: 레고 스타워즈 세트    │
│                                     │
│  ████████████░░░░░  750 / 1000     │
│                                     │
│  🏅 Large 목표                      │
│  📅 예상 달성: 약 3주 후             │
│                                     │
│  [포인트 넣기]                       │
└─────────────────────────────────────┘
```

#### 목표 변경 정책 (V1.1 수정)

**V1.0:**
```
"No Moving Goalposts: 코스트/규칙 쉽게 변경 불가"
→ 현실적이지 않음 (가격 변동, 단종 등)
```

**V1.1: 투명한 변경 프로세스**

| 상황 | 처리 방식 |
|------|----------|
| 목표 가격 상승 | 기존 모은 포인트 100% 유지, 차이만 추가 필요 |
| 목표 단종/불가 | 모은 포인트 100% 반환, 새 목표로 이전 옵션 |
| 부모 임의 변경 | 아이에게 사유 설명 필수 + 변경 로그 기록 |

**핵심:** "변경 불가"가 아니라 **"변경 시 투명한 커뮤니케이션"**

## 4. 아이 화면 UI 장치 (3종)

```
⭐⭐⭐ = Large (큰 노력)
```

또는 아이콘:
```
🌱 = Small
🌿 = Medium
🌳 = Large
🏔️ = XL
```

## 5. 상태 표시 시스템

### 5.1 아이 관점 3가지 상태

모든 리워드는 아래 3가지 중 하나로만 표시:

| 상태 | 아이콘 | 설명 |
|------|--------|------|
| **바로 사용** | ✅ | 구매 즉시 효력 |

### 5.2 신뢰 핵심 규칙

| 상황 | 처리 |
|------|------|
| 승인 대기 48시간 초과 | **자동 100% 환불** |
| 부모 거절 | **즉시 100% 환불 + 사유 표시** |
| 예약 후 일정 취소 | **자동 100% 환불** |

**V1.1 원칙:** 아이에게 불리한 "일부 환불" 페널티 없음

## 6. Anti-Exploitation & Psychological Safety

### 6.1 Top 10 Rules (V1.1 필수)

| # | 규칙 | 설명 |
|---|------|------|
| 1 | **Sibling Firewall** | 형제 간 잔액/구매내역 비공개 |
| 2 | **Transparent Changes** | 목표 변경 시 사유 설명 + 포인트 100% 보전 |
| 3 | **Screen Budget System** | 주간 예산 + 일일 상한 + 쿨다운 |
| 4 | **Request Cooldown** | 반복 요청 시 버튼 일시 비활성화 |
| 5 | **No Debt** | 마이너스 잔액, 외상 금지 |
| 6 | **Structured Rejection** | 거절 시 사유 템플릿 + 자동 환불 |
| 7 | **Kindness Separation** | 기본적 친절/배려는 포인트화 금지 |
| 8 | **Change Log** | 부모 포인트 조정 시 아이에게 사유 표시 |
| 9 | **Hidden Exchange Rate** | 환율은 숨기고, 배지/티어/게이지로만 체감 |
| 10 | **Limited Admin** | 부모는 흐름 처리자, 중앙은행 아님 |

### 6.2 Kindness 분리 기준 (V1.1 명확화)

| 행동 유형 | 포인트 | 이유 |
|----------|--------|------|
| **가족 내 기본 배려** | ❌ 금지 | 가족 역할의 일부 |
| (예: 식탁 정리 돕기, 동생에게 양보) | | |
| **추가적 책임/시간 투자** | ✅ 가능 | 기대 이상의 노력 |
| (예: 2시간 이상 동생 돌봄, 큰 집안일) | | |
| **외부/사회적 친절** | ❌ 금지 | 내적 동기 유지 |
| (예: 친구 도움, 봉사활동) | | |

## 7. 부모 운영툴

### 7.1 설계 원칙

**"부모는 중앙은행이 아니라 환경 설계자"**

부모가 할 수 있는 일:
- ✅ 승인/거절 (사유 템플릿 사용)
- ✅ 예약 리워드 일정 확정
- ✅ 주간 예산/상한 설정
- ✅ 템플릿 보너스 지급 (서프라이즈, 회복 등)
- ❌ 자유로운 포인트 증감 (제한됨)

### 7.2 포인트 조정 템플릿

| 템플릿 | 용도 | 범위 |
|--------|------|------|
| **서프라이즈 보너스** | 특별히 잘했을 때 | +20~50p |
| **회복 보너스** | 아팠거나 힘들었을 때 | +30~50p |
| **가족 이벤트 보너스** | 생일, 명절 등 | +50~100p |
| **실수 조정** | 부모 입력 실수 수정 | ±자유 (로그 필수) |

### 7.3 Policy Lock (정책 잠금)

## 9. V1.1 요약 (한 문단)

EarnQuest는 포인트를 **게임 토큰(선택 티켓)**으로 설계하며, 아이에게는 노력 배지 + 티어 + 예산 게이지로 공정성을 체감하게 하고, **부모에게는 가치 컨텍스트 패널로 일관된 가격 책정을 지원**한다. Screen Time은 "구매"가 아닌 **"기본 예산 + 보너스 획득" 시스템**으로 정의하여 환율 노출 모순을 해결했다. V1.1에서는 카테고리를 Screen/Rewards/Goals 3개로 단순화하고, 미정의였던 Trust 시스템을 제거하여 실행 가능성을 높였다. 모든 리워드는 "즉시/승인필요" 두 상태로만 구분하며, Experience형은 예약 요청권으로서 미응답/거절 시 자동 100% 환불한다. Goals의 가격 변경은 **투명한 커뮤니케이션 프로세스**를 통해 처리하며 기존 포인트는 100% 보전한다. 형제 비교·부채·기본 친절의 포인트화는 금지하고, 부모 운영툴은 승인/일정/예산/템플릿 보너스 중심으로 제한하여 중앙은행 역할을 최소화한다.
